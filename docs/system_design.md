# 要件定義書 (v2.2)

> **更新日**: 2026年6月28日  
> **ベース**: `docs/202606_spec4/direction.md` + `docs/202606_spec3/screen_spec.md`（v2.1）+ `docs/202606_spec2/screen_spec.md`

## 1. プロジェクト概要・背景

本プロジェクトは、名著『ずっとやりたかったことを、やりなさい（The Artist's Way）』の12週間プログラムをWeb上で継続的にサポートするためのアプリケーションです。ユーザーは毎日の「モーニングページ」と毎週の「アーティストデート」を記録し、週次の進捗を確認できるようになります。

- 目的: 12週間の実践を習慣化し、振り返りと自己管理を補助すること
- メインユーザー: The Artist's Wayのプログラムを実践したい個人ユーザー
- プラットフォーム: React フロントエンド + Cloudflare Workers バックエンド + Cloudflare D1 / R2 ストレージ
- 認証基盤: Supabase Auth

### v2.2の主な追加・変更（spec4）
- **利用規約の同意必須化**: サインアップ画面に「利用規約およびプライバシーポリシーに同意する」チェックボックスを追加し、未同意時は登録ボタンを非活性化
- **プログラム完遂演出**: 12週間完遂（Day 84 到達）時に祝福モーダル（紙吹雪演出）を表示
- **ホームUI改善**: 「本日: Day X / 84」表記、モーニングページ進捗の「X/7」テキスト表示、週移動ボタンの視認性向上、「現在の週に戻る」ボタン、保存完了トースト通知
- **統計UI改善**: 日次データの単一選択＋「詳細」（`/morning-page?date=` へ遷移）、アーティストデート軌跡の 〇/✕ 表記

### v2.1の主な追加・変更（spec3）
- **アカウント作成（サインアップ）**: ログイン画面でログイン / 新規アカウント作成を切り替え可能に
- **全体進捗インジケーター**: ホームに「Day XX / 84」を表示
- **dayX クリック編集**: ホームの各日ドットから対象日のモーニングページ編集へ遷移（過去日は確認ダイアログ経由）
- **モーニングページ保存後の自動リダイレクト**: 保存成功時に `/home` へ遷移
- **日次ステータス同期**: モーニングページを保存（書き込み）した日はステータスを「完了」に更新（v2.0 の同期方針を踏襲）

### v2.0の主な新機能
- **統計画面**: 12週間の軌跡を可視化（グラフ、ヒートマップ）
- **タイムトラベル機能**: 過去・現在・未来の週を自由に閲覧
- **日次ステータス表示**: 7日間のドット表示で直感的な進捗把握
- **視覚的演出**: 時間軸に応じた色調変化
- **改善されたUX**: 保存後の自動遷移、ステータス同期の最適化

## 2. ターゲットユーザーと提供価値

### ターゲットユーザー
- The Artist's Wayを実践したいが紙ベース管理が負担になる人
- 習慣化のために進捗をWebで一元管理したい人
- 日々の思考と週次のクリエイティブ活動を簡単に記録したい人

### 提供価値
- 毎日のモーニングページの入力・保存を簡潔に行えること
- 週次のアーティストデート実施状況を記録し、週をまたいだ継続性を伝えること
- 現在の週次進捗をひと目で確認でき、継続意欲を高めること
- 著作権上問題のある書籍内容は一切表示せず、安全にプログラムの枠組みだけをサポートすること

#### UI要件

### 3.2 ログイン・アカウント作成画面（v2.2 改修）

#### 表示内容
- ログイン / 新規アカウント作成のモード切り替えタブ
- ログインフォーム（メール / パスワード、Google OIDC 等）
- アカウント作成フォーム（メール / パスワード / パスワード確認）
- **利用規約・プライバシーポリシー同意チェックボックス（v2.2 新規）**: サインアップモード時に「利用規約およびプライバシーポリシーに同意する」チェックボックスを表示
- ログイン・サインアップ成功後、ユーザーホームへ遷移

#### 挙動
- 未認証ユーザーはこの画面以外の画面にアクセス不可
- 認証基盤は Supabase Auth を利用し、クライアントは Supabase のセッション/アクセストークンでログインする
- **サインアップ**: クライアントバリデーション（メール形式、パスワード一致、**利用規約同意**）の通過後、Supabase Auth のサインアップAPIを呼び出す。成功時は確認メール案内表示、または自動ログインして `/home` へ遷移
- **利用規約の同意必須化（v2.2 新規）**: 同意チェックボックスが未チェックの間は登録ボタンを非活性（disabled）にする。未チェックのまま押下された場合はバリデーションエラー（「利用規約への同意が必要です」）を表示して処理をブロックする
- 認証状態はフロントエンドで保持し、APIはCloudflare Workers 上で Supabase JWT を検証してユーザー識別子を取得する
- ログイン・サインアップエラー時はエラー文言を表示（「既に登録されているメールアドレスです」「パスワードが一致しません」「利用規約への同意が必要です」等）

### 3.3 ユーザーホーム画面（大幅改修）

#### 表示内容
- **全体進捗インジケーター**: 「本日: Day X / 84」（プログラム開始日を起点とした経過日数）を画面上部に表示（v2.2 で「本日:」ラベルを付与）
- **週次タイムトラベル機能**: 左右スワイプまたは矢印ボタンで第1週〜第12週を切り替え
- **ナビゲーションの視認性向上（v2.2 新規）**: 週移動ボタン（`<` / `>`）の可視性を高める。「現在の週に戻る」ボタンを過去/未来週の閲覧中のみプライマリカラーで表示
- 現在の週次表示: 「現在第◯週目」（初期表示は現在の経過週）
- **日次ステータス表示**: 7日間のドットグリッド（Day 1〜Day 7、クリック可能）。今週の達成日数を「X/7」テキストで併記（v2.2 新規）
- **保存完了トースト通知（v2.2 新規）**: `/morning-page` 保存後の `/home` リダイレクト直後に「モーニングページを保存しました」を数秒間表示
- **プログラム完遂モーダル（v2.2 新規）**: 経過日数が Day 84 に到達した場合、ログイン直後/ホーム遷移時に祝福モーダル（紙吹雪演出）を表示
- モーニングページへのリンク
- アーティストデートへのリンク
- 各タスクのチェックリスト表示（今日の入力状況 / 今週の実施状況）
- 統計画面へのリンク（共通ナビゲーション）

#### 詳細仕様

##### 週次タイムトラベル機能（新規）
- ユーザーの開始日と現在のシステム日付から「現在の経過週」を算出し、初期表示とする
- 左右の矢印ボタン「＜」「＞」または画面スワイプで、第1週〜第12週を切り替え可能
- 表示中の週番号を状態管理し、APIパラメータとして送信（`GET /api/progress?week_number=N`）

##### 視覚的演出（新規）
- **過去の週を表示中**: 
  - セピアまたはモノトーン（グレースケール）のスタイル適用
  - 入力ボタンを非活性化（閲覧のみモード）
- **未来の週を表示中**: 
  - 半透明オーバーレイまたはソフトブルーグラデーション適用
  - タスクをロック状態で表示
- **現在週を表示中**: 
  - 通常ブランドカラー（高コントラスト、アクティブ表示）

##### 日次ステータス表示（新規）
- 7日間のドットグリッド（Day 1〜Day 7）を横一列に配置
- **実行済みの日**: ターコイズグリーン系の鮮やかなカラー、チェックマーク付き
- **未実行の日**: グレーアウト、または破線の枠線のみ
- 各ドットの下部に文字数を小さく表示
- 当日のドットまたはボタンから `/morning-page` へ遷移

##### dayX クリック編集・確認ダイアログ（v2.1 新規）
- 各ドット（`dayX`）はクリック可能なボタンとし、クリックで `/morning-page?date=YYYY-MM-DD` へ遷移
- クリックした日付が**本日より過去**の場合、「過去のページです。編集を開始してよろしいですか？」の確認ダイアログを表示し、OK時のみ遷移（キャンセル時はホームに留まる）
- 未来日のドットはロック状態とし、クリック遷移は不可

##### 編集後のステータス同期（書き込み時に「完了」へ更新）
- **モーニングページを保存（書き込み）した日は、日次ステータス（色・チェック）を「完了」に更新する**（新規作成・過去日編集いずれの保存も対象）
- 保存成功時はキャッシュ無効化または Progress / MorningPages API の再フェッチを行い、「完了」ステータスと文字数表示を遅延なく最新DB状態に反映する
- バックエンド側でも `MorningPages` 保存時に該当週の `Progress.morning_page_done` を即座に評価・更新する

##### ホームUI改善（v2.2 新規）
- **現在日数の明記**: 画面上部に「本日: Day X / 84」を表示し、プログラム開始日からの経過日数を明確にする
- **モーニングページ進捗の「X/7」表記**: 現在の週におけるモーニングページの達成日数を分母・分子のテキスト（例：3/7）で直感的に表示する
- **ナビゲーションの視認性向上**:
  - 週移動ボタン（`<` / `>`）のクリック領域・サイズを広げて可視性を高める
  - 「現在の週に戻る」ボタンは、現在の経過週**以外**を閲覧している間のみプライマリカラーで表示し、押下で現在週に戻す
- **保存完了フィードバック（トースト通知）**: モーニングページ保存後の `/home` リダイレクト直後に「モーニングページを保存しました」のトースト/スナックバーを画面下部または上部に数秒間表示し、フェードアウトさせる

##### プログラム完遂時の祝福モーダル（v2.2 新規）
- 経過日数が Day 84 に到達（12週間プログラム完遂）したとシステムが判定した場合、ログイン直後または `/home` 遷移時に祝福モーダルを表示する
- 「12週間のプログラム完遂、おめでとうございます！」のメッセージと紙吹雪（Confetti）演出で達成感を高める
- 判定基準は `GET /api/progress` の `current_day >= 84`。本バージョンでは到達後ログインのたびに表示する（表示済みフラグの保存は行わない）

##### 従来の機能
- 週番号はユーザーの最初の開始日または最終入力日から算出
- モーニングページリンクは当日入力ページへ遷移
- アーティストデートリンクは今週のチェック入力ページへ遷移
- チェックリストには次の項目を表示
  - 「モーニングページ入力済み」(当日入力が存在する場合)
  - 「アーティストデート完了」(今週分のチェックが保存済みの場合)

### 3.4 モーニングページ画面（v2.1 改修）

#### 表示内容
- 対象日タイトル（「Day X のモーニングページ」/ 日付）。`?date=YYYY-MM-DD` 指定時はその日付、未指定時は当日
- テキスト入力欄（マルチライン）
- リアルタイム文字数カウント
- 保存ボタン
- エラー通知（保存失敗時）

#### 文字数カウントのルール
- 入力欄内の全テキスト文字数をリアルタイムでカウント
- 改行も1文字としてカウント
- 空白文字もカウント対象
- カウントは保存ボタン押下前後で常時同期
- 0文字の場合は「0文字」と表示

#### 保存機能
- 保存ボタン押下でバックエンドAPIへPOST送信
- 保存対象: user_id, 日付（`date` クエリまたは当日）, テキスト内容, 文字数
- **保存成功時の挙動変更（v2.1 新規）**: 書き込みが正常に完了し `200 OK` が返却された場合、画面に留まらず**自動的に `/home` へリダイレクト**
- 保存失敗時は「保存に失敗しました。再度お試しください。」を表示し、画面遷移せずに再試行を促す

#### 入力バリデーション
- テキストが空の場合、保存時に警告を表示し保存不可
- 最大文字数制限は現状未定。必要に応じ後続で設計

### 3.5 アーティストデート画面

#### 表示内容
- 「今週出かけたか」チェックボックス
- 「わくわくしたか」チェックボックス
- 保存ボタン
- 直近保存済みのステータス表示

#### チェックボックス仕様
- 2つのフラグは Boolean 形式
- 具体的な文章入力は行わない
- UI上ではそれぞれ次のように表示
  - 「今週、アーティストデートに出かけた」
  - 「今週、わくわくした出来事があった」

#### 保存機能
- 保存ボタン押下でバックエンドAPIへPOST送信
- 保存対象: user_id, week_number, went_out(bool), excited(bool), 保存日時
- **保存成功時の挙動変更（新規）**: データベースへの書き込みが正常に完了し `200 OK` が返却された場合、**自動的に `/home` へリダイレクト**
- 保存失敗時は「保存に失敗しました。再度お試しください。」を表示し、画面遷移せずに再試行を促す

#### 入力バリデーション
- どちらも未選択でも保存可能とする
- ただし保存後はチェック済み/未チェックが正しく反映されること

### 3.6 統計画面（新規）

#### 表示内容
ユーザーが12週間のプログラムにおいて、どのような軌跡をたどってきたかを定量的に振り返るための画面です。

##### 1. 12週間総合サマリーカード
- モーニングページの総執筆日数（例：52日 / 84日中）
- 累計総文字数（例：68,500 文字）
- アーティストデート累計達成率（例：8週 / 12週中）

##### 2. 12週間の執筆文字数推移グラフ
- 横軸: Week 1〜Week 12
- 縦軸: 各週の合計文字数または平均文字数
- 棒グラフまたはラインチャートで表示
- 未到達の未来週は「未到達（Pending）」または非表示

##### 3. 日次アクティビティヒートマップ
- GitHub草グラフ風の12週間×7日間（84マス）のマトリクス
- モーニングページ実行日は文字数に応じて濃い色で表示
- 未実行日は空白/薄いグレーで表示
- マウスホバーで詳細（日付・文字数）を表示

##### 3-2. 日次データの選択と詳細確認（v2.2 新規）
- カレンダー/リスト表示の各日を、ラジオボタンのように単一選択（アクティブ状態を切り替え可能）できる
- 日付を選択すると「詳細」ボタンが出現し、押下で該当日のモーニングページ内容を確認できる
- 詳細は既存の `/morning-page?date=YYYY-MM-DD` 画面へ遷移して閲覧・編集する（過去日はモーニングページ画面側のルールに従う）

##### 4. アーティストデート軌跡（v2.2 表記調整）
- 第1週〜第12週の実施状況を直感的な記号で一覧表示
- 実施した週は「〇（マル / アクセントカラー）」、未実施の週は「✕（バツ / グレー）」で表示し、ひと目で達成状況を把握できるようにする（`went_out` / `excited` をそれぞれ 〇/✕ で表現）

#### 技術仕様
- **API**: `GET /api/statistics`
- **グラフライブラリ**: Recharts または Chart.js を使用
- **データ集計**: バックエンド側で週ごとの集計クエリ（`GROUP BY week_number`）を実行

## 4. 非機能要件

### 著作権への配慮
- アプリ内に書籍本文や具体的な引用を掲載しない
- 週次の表示は「第◯週目」など抽象的な表現にとどめる
- UI文言は独自説明で補完し、書籍コンテンツを再現しない

### セキュリティ
- 認証済みユーザーだけが自分のデータにアクセスできる
- 認証基盤は Supabase Auth を利用し、APIはCloudflare Workers上で Supabase の JWT / セッションを検証する
- Cloudflare Workersでの認証チェックを必須化し、D1やR2へのアクセスはバックエンド経由のみとする
- ユーザーデータは Supabase の認証IDとアプリケーションのユーザーIDを紐付けて管理する
- セッション固定化やCSRF対策を検討する
- ローカル検証にはMiniflareコンテナを使用し、Cloudflare Workersの動作を再現した環境でテストする

### プライバシー
- モーニングページは個人的な思考の記録となるため、ユーザー以外からのアクセスを禁止
- 可能であればデータは暗号化またはアクセス制限されたストレージで保存
- 認証情報・個人情報は最小限に留める

### パフォーマンス
- フロントエンドはSPAとして高速表示
- 文字数カウントはクライアントサイドで即時反応
- 保存APIは常に最短で応答し、UIに保存結果を速やかに反映
- ロード時は必要なデータのみ取得し、不要な全件フェッチは行わない

## 5. 画面遷移図（テキスト表現）

1. LP
   - [ログインへ] → ログイン画面

2. ログイン・アカウント作成画面
   - ログイン認証成功 → ユーザーホーム
   - [利用規約同意チェック] → 新規アカウント作成成功 → 確認メール案内 / 自動ログイン → ユーザーホーム
     （同意チェックが無い間は登録ボタン非活性）
   - 認証・登録失敗 → エラー表示

3. ユーザーホーム
   - [当日 dayX クリック] → モーニングページ画面（`?date=` 当日）
   - [過去 dayX クリック] → 確認ダイアログ「OK」→ モーニングページ画面（`?date=` 該当日）
   - [モーニングページ] → モーニングページ画面
   - [アーティストデート] → アーティストデート画面
   - [統計] → 統計画面（新規）
   - [＜＞ボタン] → 週次切り替え（タイムトラベル）
   - [現在の週に戻る]（過去/未来週の閲覧中のみ表示）→ 現在の経過週へ戻る
   - ※ Day 84 到達時、ログイン直後/ホーム遷移時に祝福モーダル（紙吹雪）を表示
   - ※ モーニングページ保存後のリダイレクト直後に保存完了トーストを表示

4. モーニングページ画面
   - [保存] → API保存 → **成功時は自動的に `/home` へリダイレクト**（失敗時は遷移せずエラー表示）
   - [ホームへ] → ユーザーホーム
   - ※ 保存に成功した日は、ホームの該当 Day ステータスが「完了」に更新される

5. アーティストデート画面
   - [保存] → API保存 → **成功時は自動的に `/home` へリダイレクト**
   - [ホームへ] → ユーザーホーム

6. 統計画面（新規）
   - 12週間の総合サマリー、グラフ、ヒートマップを表示
   - [日付を選択 → 詳細] → モーニングページ画面（`?date=` 該当日）
   - [ホームへ] → ユーザーホーム

## 6. データモデル・テーブル定義案

### Users
- id: TEXT PRIMARY KEY
- supabase_user_id: TEXT UNIQUE NOT NULL
- email: TEXT NOT NULL
- created_at: DATETIME NOT NULL
- last_active_at: DATETIME NULL
- auth_provider: TEXT NULL

### MorningPages
- id: INTEGER PRIMARY KEY AUTOINCREMENT
- user_id: TEXT NOT NULL
- entry_date: DATE NOT NULL
- content: TEXT NOT NULL
- character_count: INTEGER NOT NULL
- created_at: DATETIME NOT NULL
- updated_at: DATETIME NOT NULL

### ArtistDates
- id: INTEGER PRIMARY KEY AUTOINCREMENT
- user_id: TEXT NOT NULL
- week_number: INTEGER NOT NULL
- went_out: BOOLEAN NOT NULL DEFAULT 0
- excited: BOOLEAN NOT NULL DEFAULT 0
- created_at: DATETIME NOT NULL
- updated_at: DATETIME NOT NULL

### Progress
- id: INTEGER PRIMARY KEY AUTOINCREMENT
- user_id: TEXT NOT NULL
- week_number: INTEGER NOT NULL
- morning_page_done: BOOLEAN NOT NULL DEFAULT 0
- artist_date_done: BOOLEAN NOT NULL DEFAULT 0
- updated_at: DATETIME NOT NULL

#### 補足
- `user_id`は`Users.id`に外部キーとして紐づく
- `week_number`は 1 〜 12 の範囲を想定
- `MorningPages.entry_date`は日次記録の重複を防ぐためユニーク制約を検討
- `ArtistDates.week_number`は週次ログの重複を防ぐためユニーク制約を検討

## 8. 開発上の注意点

- Docker 開発環境では `docker compose up --build` で `workers` と `web` を同時起動し、`8787` と `5173` のポート競合に注意する。
- Workers のローカル開発では `wrangler dev --ip 0.0.0.0 --port 8787` と `wrangler.toml` の `dev.ip` を使い、コンテナ内外のアクセスを両立させる。
- `.dockerignore` に `.wrangler`, `node_modules`, `.git`, `gen_logs` などを含め、Docker ビルドコンテキストを最小化する。
- D1 の接続情報は `wrangler.toml` や環境変数で管理し、直接ソースコードに埋め込まない。
- Workers 側 API は必ずユーザー認証と `user_id` 紐付けを行い、認証なしのアクセスや他ユーザーのアクセスを防ぐ。
- フロントエンドは SPA として構築し、未認証ユーザーの保護ページアクセス時は `/login` にリダイレクトする。
- モーニングページ / アーティストデートの保存処理は、ローカルでは JSON API レスポンスとエラー処理の動作を確認しておく。
- 本アプリは書籍コンテンツを扱わないため、開発時にも著作権に抵触するような文言や本文を避ける。

## 7. バックエンドAPI拡張（v2.0）

### 7.1 `GET /api/progress` の拡張
- `week_number` パラメータを受け取り、指定週の詳細データを返却
- レスポンスに以下を含む:
  - 週番号（`week_number`）
  - 現在の経過週（`current_week`）
  - 今週のモーニングページ実行日数（`morning_pages_this_week`）
  - 7日間の日次ステータス配列（`daily_status`）
    - 各日: `day`, `date`, `done`, `character_count`
  - アーティストデート状態（`artist_date`）
    - `went_out`, `excited`

### 7.2 `GET /api/statistics` の新設
- ユーザーの全データを集計して返却
- レスポンス内容:
  - **summary**: 総執筆日数、累計文字数、アーティストデート達成週数
  - **weekly_stats**: 週ごとの文字数集計（`GROUP BY week_number`）
  - **daily_activity**: 84日分の日次データ（日付、文字数、週番号、曜日）
  - **artist_date_history**: 週次のアーティストデート達成状況

### 7.3 自動ステータス更新
- `POST /api/morning-pages` 保存時に、該当週の `Progress.morning_page_done` を自動更新
- `POST /api/artist-dates` 保存時に、該当週の `Progress.artist_date_done` を自動更新
