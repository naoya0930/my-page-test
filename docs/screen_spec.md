## 2. 画面遷移図（フロー）

### 未認証ユーザーのアクセス
- どのアドレスにアクセスしても、`/` または `/login` 以外はフロントエンドで捕捉し、`/login` へリダイレクト。

### ログイン / アカウント作成フロー
- `/login`（ログインタブ）で認証成功 ➔ `/home` へ遷移。
- `/login`（新規アカウント作成タブ）で「利用規約およびプライバシーポリシーに同意する」チェック ➔ サインアップ成功 ➔ 確認メール案内表示、または自動ログインして `/home` へ遷移（v2.3：同意チェックが無い間は登録ボタンを非活性）。

### ホームからの遷移
- `/home` ➔ [当日の dayX クリック] ➔ `/morning-page?date=YYYY-MM-DD`
- `/home` ➔ [過去日の dayX クリック] ➔ 確認ダイアログで「OK」➔ `/morning-page?date=YYYY-MM-DD`
- `/home` ➔ [アーティストデート入力] ➔ `/artist-date`
- `/home` ➔ [統計を見る] ➔ `/statistics`
- `/home` ➔ [「現在の週に戻る」ボタン]（過去/未来週の閲覧中のみ表示）➔ 現在の経過週へ戻る（v2.3）。
- `/home`（過去週を閲覧中）➔ [「過去の記録を閲覧できます」リンク] ➔ `/statistics`（v2.4：案内テキストをクリッカブルなリンク化）。
- `/statistics` ➔ [日付を選択 ➔ 詳細] ➔ `/morning-page?date=YYYY-MM-DD`（v2.3）。

### プログラム完遂時のフロー（v2.3 新規）
- ログイン直後または `/home` 遷移時、**経過日数が Day 84 に到達している**と判定された場合、祝福モーダル（紙吹雪演出付き）を表示する。
- ※ 本バージョンでは、到達後はログインのたびに表示する（表示済みフラグの保存は行わない）。

### 本日のハイライト・達成セレブレーションのフロー（v2.4 新規）
- `/home`（現在週を表示中）➔ 「今週の実行状況」の**本日**に該当する要素を強調（バッジ「今日」＋リング）。
- `/home` ➔ **モーニングページ 7/7** または **アーティストデート完了**（OR）で、該当**カードに祝福スタイル**（薄ゴールド背景＋豪華アイコン）を適用。達成状態が続く限りホーム表示のたびに適用する（状態ベース・派手なアニメーションなし）。
- `/statistics` ➔ 日次アクティビティヒートマップ上で**本日のセル**を枠線アクセント＋ツールチップ「今日」で強調。
- ※ いずれも「本日」判定はクライアントのシステム日付と各セルの日付の突合で行う（新規API不要）。

### タスク完了後のフロー
- `/morning-page` ➔ [保存] ➔ 成功時（`200 OK`）：**自動的に `/home` へリダイレクト**する（v2.2 仕様変更）。リダイレクト直後、`/home` 上に「モーニングページを保存しました」のトースト通知を数秒間表示しフェードアウトさせる（v2.3）。失敗時は遷移せずエラー表示。
- `/artist-date` ➔ [保存] ➔ 成功時（安全に保管された場合）：**自動的に `/home` へ遷移する**（追加要件）
- ※ モーニングページを保存した日は、ホーム画面の該当 Day の日次ステータスが「完了」に更新される（詳細は 3.3-④ を参照）。

### 共通ナビゲーション
- 認証後の全画面において、ヘッダー等の共通コンポーネントから `/home` および `/statistics` への相互遷移が可能。

### 3.2 ログイン・アカウント作成画面（v2.3 改修）

**パス**: `/login`  
**認証**: 不要（認証済みなら `/home` へリダイレクト）  
**ファイル**: `src/pages/LoginPage.tsx`

#### 表示内容
- **認証モード切り替えタブ / リンク**: 「ログイン」と「新規アカウント作成」を切り替えるUI。
- **ログインフォーム**: メールアドレス入力欄、パスワード入力欄、「ログイン」ボタン。ソーシャルログイン（Google OIDC）も併設可。
- **アカウント作成フォーム（新規追加）**: メールアドレス入力欄、パスワード入力欄、パスワード確認用入力欄、「アカウントを作成する」ボタン。
- **利用規約・プライバシーポリシー同意チェックボックス（v2.3 新規）**: アカウント作成（サインアップ）モード時のみ表示。「利用規約およびプライバシーポリシーに同意する」のチェックボックス。
- エラーメッセージ表示エリア（フォーム下部）。

#### 挙動・セキュリティ
- 認証基盤は `Supabase Auth` を利用。
- **ログイン処理**: 認証成功時、アクセストークンをフロントエンド（状態/ストレージ）に保持し、`/home` へ自動遷移。
- **アカウント作成（サインアップ）処理**:
  - **利用規約の同意必須化（v2.3 新規）**: 同意チェックボックスが**未チェックの間は登録ボタン（「アカウントを作成する」）を非活性（disabled）**にする。あわせて、未チェックのまま押下された場合に「利用規約への同意が必要です」というバリデーションエラーを表示して処理をブロックする（二重防御）。

  - 登録成功時は、確認メール送信の案内を表示するか、または自動ログイン処理を行い `/home` へ遷移させる。
- エラー発生時は、フォーム下部に赤字で具体的なエラー内容（「メールアドレスまたはパスワードが違います」「既に登録されているメールアドレスです」「パスワードが一致しません」「利用規約への同意が必要です」等）を即時表示。

### 3.3 ユーザーホーム画面

**パス**: `/home`  
**認証**: 必要（未認証なら `/login` へリダイレクト）  
**ファイル**: `src/pages/HomePage.tsx`  
**API**: `GET /api/progress?week_number=N`

ユーザーがプログラムの進捗を確認し、その日のアクションを起こすための中心的な画面です。

#### ⓪ 全体進捗インジケーター「本日: Day X / 84」（v2.3 表記調整）
- 画面上部の目立つ位置に「**本日: Day X / 84**」のように、プログラム開始日を起点とした現在の経過日数（例：Day 15）を明確に表示する。
- 表示週の切り替え（タイムトラベル）とは独立して、常に「本人の経過日数」を基準に算出する。
- ※ v2.2 までの「Day XX / 84」表記に「**本日:**」のラベルを付与し、本日が Day いくつなのかをより直感的に伝える（spec4）。

#### ① 週次表示とタイムトラベル（左右スワイプ）機能（追加要件）

##### 初期表示
- アクセス時、ユーザーの開始日と現在のシステム日付から算出された**「現在の自身の経過週（例：第3週目）」**をデフォルトで中央に表示する。

##### 左右スワイプ / ナビゲーション
- 画面を左右にスワイプ（または左右の矢印ボタン「＜」「＞」をクリック）することで、過去の週（第1週〜現在未満）や未来の週（現在より上〜第12週まで）の状態に表示を切り替えられる。

##### ナビゲーションの視認性向上（v2.3 新規）
- **週移動ボタン（`<` / `>`）**: タップ/クリック領域とサイズを広げ、可視性を高める。
- **「現在の週に戻る」ボタン（v2.3 新規）**: 現在の経過週**以外**（過去週・未来週）を閲覧している際**にのみ**表示する。プライマリカラー（目立つ色）を使用し、押下すると現在の経過週の表示に即座に戻す。現在週を表示中は非表示とする。

##### 「いまではないもの」を見ている際の視覚的演出（追加要件）
- 現在の表示週が「現在の本人の経過週」と異なる場合、UIの色調を明確に変更し、過去や未来のデータを見ている（タイムトラベル中である）ことをユーザーに認識させる。

**演出案:**
- **過去の週を表示中:** 画面全体の色彩を「セピア」または「モノトーン（グレースケール）」に変更。各入力ボタンは非活性（閲覧のみの挙動）とする。
  - **（v2.4 変更）「(閲覧のみ)」ラベルの削除**: 過去週表示時に表示していた「**(閲覧のみ)**」というテキストは**表示しない（削除）**。UIをクリーンに保つ。入力ボタンを非活性にする操作制御自体は従来通り維持し、**削除するのはラベルテキストのみ**とする。
  - **（v2.4 新規）案内テキストの統計画面へのリンク化**: 過去週表示時の「**過去の記録を閲覧できます**」というテキストを、クリッカブルなテキストリンク（またはボタン）に変更する。押下で `/statistics`（統計画面）へ遷移するルーティングを設定し、リンクと分かるスタイル（下線 / プライマリカラー等）を付与する。
- **未来の週を表示中:** 画面全体に「半透明のオーバーレイ」または「ソフトブルーのグラデーション」を適用し、未来のプレースホルダーであることを明示。タスクはロック状態とする。
- **現在週を表示中:** アプリケーションの通常ブランドカラー（高コントラスト、アクティブ表示）で表示。

#### ② モーニングページ・日次実行ステータス（追加要件）

##### 表示デザイン
- 今週（表示中の週）の7日間において、何日目に実行し、何日目に実行していないのかがひと目で直感的にわかるデザインを採用する。
- **達成状況の分母・分子テキスト表示（v2.3 新規）**: 現在の週におけるモーニングページの達成日数を「**X/7**」という分母・分子のテキスト形式で併記する（例：今週3日書いた場合は「3/7」）。ドットグリッドの近傍（見出し横など）に配置し、直感的に進捗が把握できるようにする。
- **本日の要素のハイライト（v2.4 新規）**: 「今週の実行状況」のドットグリッドにおいて、**現在の日付（本日）に該当する曜日・日付の要素を強調**する。本日のドット/セルに「**今日**」を示す小さなバッジを付与し、あわせて枠線（リング）をプライマリカラーで強調する。達成状態（実行済み/未実行）の色表現とは独立して重畳する。
  - **判定**: クライアントのシステム日付と `daily_status[].date` を突合し、一致する Day を「本日」とする（新規API不要）。
  - 過去週・未来週を閲覧中（タイムトラベル中）は、その週に本日が含まれないため強調は表示されない（現在週を表示している場合のみ本日が出現する）。
- **「未完了」ラベルの非表示（v2.4 新規）**: 従来「モーニングページ」に表示していた「**未完了**」というテキストは**表示しない（削除）**。達成/未達成の状態はドットの色や「X/7」表記で表現済みのため、冗長なラベルは設けない。

**デザイン案（ドットグリッド / カレンダーマトリクス）:**
- 横一列に7つの円（ドット）またはカードを配置（Day 1 〜 Day 7）。
- **実行済みの日の表示:** 鮮やかな成功カラー（例: ターコイズグリーン、チェックマーク付きの塗りつぶし）で表示。
- **未実行（かつ過去・当日）の日の表示:** グレーアウト、または破線の枠線のみで表現し、スキップされたか未着手であることを明示。
- 各ドットの下部には、その日に執筆した「文字数」を小さく表示する。
- 各ドット（`dayX`）はクリック可能なボタンとして機能する。

##### dayX クリック時の編集機能（v2.2 新規追加）
- グリッド内の任意の `dayX` をクリックすると、その日付に対応するモーニングページ編集画面（`/morning-page?date=YYYY-MM-DD`）へ遷移する。
- 当日のドット、または「今日のモーニングページを書く」ボタンからも `/morning-page`（当日）へ遷移可能。

##### 過去ページの編集確認アラート（v2.2 新規追加）
- クリックした `dayX` が**本日より過去の日付**である場合、「過去のページです。編集を開始してよろしいですか？」という確認ダイアログ（ブラウザの `confirm`、またはカスタムモーダル）を表示する。
- 「OK / 同意」を選択した場合のみ `/morning-page?date=YYYY-MM-DD` への遷移を許可し、「キャンセル」した場合はホーム画面に留まる。
- 未来日のドットはロック状態（①の演出に準拠）とし、クリックによる遷移は不可。

#### ③ アーティストデート進捗表示

##### 表示内容
- 今週分の実施状況をチェックリスト形式で表示。
  - 「今週、アーティストデートに出かけた」
  - 「今週、わくわくした出来事があった」

##### リンク
- クリックで `/artist-date` へ遷移。

#### ④ 編集後のステータス同期（書き込み時に「完了」へ更新）

##### 仕様
- **モーニングページを保存（書き込み）した日は、日次ステータス（ドットの色・チェックマーク）を「完了」に変更する。**
- 新規作成・過去日の編集いずれの保存でも、該当日のステータスを「完了」へ更新し、ドット下部の文字数表示も保存内容に応じて更新する。

##### 実装上の挙動（データ整合性）
- 保存処理が成功した段階で、フロントエンドの状態管理（React Context / SWR / TanStack Query 等）のキャッシュを無効化（Invalidate）するか、最新の Progress / MorningPages API を再フェッチ（Refetch）し、「完了」ステータスへの切り替えと文字数表示を遅延なく反映する。
- ホーム画面のコンポーネントがマウントされた際、常に最新のDB状態（該当日の `MorningPages` レコードの有無）を参照する。
- バックエンド側でも、`MorningPages` 保存時に該当週の `Progress.morning_page_done` を即座に評価・更新し、一貫したデータ返却を行う。

#### ⑤ 保存完了フィードバック（トースト通知）（v2.3 新規）
- `/morning-page` で「保存する」を実行し、`/home` にリダイレクトされた直後に、「**モーニングページを保存しました**」という一時的な通知（トースト / スナックバー）を画面下部または上部に**数秒間**表示し、フェードアウトさせる。
- 実装方法（例）: リダイレクト時にナビゲーション state（`navigate('/home', { state: { toast: '...' } })`）やクエリで保存成功を伝え、`/home` 側でトーストを表示後に state をクリアする（リロードで再表示されないようにする）。
- トーストは共通コンポーネント（6章 参照）として実装し、他画面でも再利用できるようにすることを推奨。

#### ⑥ プログラム完遂時の祝福モーダル（v2.3 新規）
- ログイン直後または `/home` 遷移時、**経過日数が Day 84 に到達している**（12週間プログラム完遂）とシステムが判定した場合、専用のモーダル（ポップアップ）を表示する。
- 「**12週間のプログラム完遂、おめでとうございます！**」といった労いのメッセージを表示し、紙吹雪（Confetti）などの視覚的な演出を取り入れて達成感を高める。
- **判定基準**: `GET /api/progress` の `current_day`（経過日数）が 84 以上であること。
- **再表示制御（本バージョンの方針）**: 到達後はログインのたびに表示する（「表示済み」フラグの保存は行わない）。将来的に「一度だけ表示」へ変更する場合は、`Users` もしくはクライアント側ストレージに表示済みフラグを持たせる拡張を検討する。
- モーダルは「閉じる」操作でホーム画面に戻れること。紙吹雪演出はアクセシビリティ配慮として `prefers-reduced-motion` 時は控えめにする。

#### ⑦ 達成時セレブレーション演出（v2.4 新規）
- **発火条件（OR・いずれか一方を満たせば発火）**:
  1. 「モーニングページ」の今週の進捗が **7/7（コンプリート）** になった。
  2. 「アーティストデート」が**完了（実施済み）**になった。
- **演出スタイル（本バージョンの方針：控えめ）**:
  - 該当条件を満たした**項目のカード単位**で演出を適用する（モーニングページのコンプリートはモーニングページのカード、アーティストデートの完了はアーティストデートのカード。両方達成時は両カードに適用）。
  - **背景色の変化**: 該当カードの背景を特別感のある色（薄いゴールド / ソフトなグラデーション）に変化させる。
  - **アイコンの変更**: チェックマークを豪華なアイコン（王冠 👑 / キラキラ ✨ / クラッカー 🎉 など）に変更する。
  - 派手なアニメーション（紙吹雪・パーティクル等）は本バージョンでは採用せず、静的な祝福スタイル（背景＋アイコン）を基本とする。
- **適用タイミング（本バージョンの方針：状態ベース・毎回適用）**:
  - 「達成した瞬間」の検出は行わず、**達成条件を満たしている限り、ホームを表示するたびに**祝福スタイルを適用する（リロード・再訪のたびに表示される）。
  - 軽微なワンポイントの動き（アイコンのポップ等）を添える場合は `prefers-reduced-motion` を尊重し、モーション低減設定時は静的表示にとどめる。
- **判定データ**: `GET /api/progress` の `morning_pages_this_week === 7`（または `daily_status` の `done` が7件）、`artist_date_done === true`（または `artist_date` の状態）。
  - ※ 「アーティストデート完了」は **`went_out`（外出したか）のみ**で判定する（確定）。`excited`（わくわくしたか）は記録するが完了判定には含めない。バックエンドの `artist_date_done`（`GET /api/progress`）および統計の `artist_date_weeks` 集計も `went_out` のみに統一する。
- ⑥ の完遂モーダル（プログラム全体の Day 84 到達）とは独立した、**今週・各タスク単位**の達成演出である。

### 3.4 モーニングページ画面（v2.2 改修）

**パス**: `/morning-page`（`?date=YYYY-MM-DD` で対象日を指定可能。未指定時は当日）  
**認証**: 必要  
**ファイル**: `src/pages/MorningPage.tsx`  
**API**: `GET /api/morning-pages?date=YYYY-MM-DD`, `POST /api/morning-pages`

#### 表示内容
- タイトル（例：「Day X のモーニングページ」または「2026年6月27日の記録」）。`date` クエリに応じて対象日を明示する。
- 広範なテキスト入力欄（マルチラインのTextArea、フォーカス時に枠線強調、オートフォーカス推奨）。
- **リアルタイム文字数カウント:** 改行、空白スペースを含むすべての入力文字数をクライアントサイドで即時カウントし、「〇〇文字」とリアルタイム表示。0文字の場合は「0文字」と表示。

- テキストが空（0文字）の場合は、保存ボタンを非活性化するか、押下時に「内容を入力してください」と警告を表示してAPI送信をブロックする。

- **保存完了後の画面遷移（v2.2 仕様変更）:**

### 3.6 統計画面（新規）

**パス**: `/statistics`  
**認証**: 必要  
**ファイル**: `src/pages/StatisticsPage.tsx`（新規作成）  
**API**: `GET /api/statistics`

ユーザーが12週間のプログラムにおいて、どのような軌跡をたどってきたか（継続の証、自己の思考パターンの変化）を定量的に振り返るための画面です。

#### 表示内容

##### 1. 12週間総合サマリーカード
- モーニングページの総執筆日数（例：52日 / 84日中）
- 累計総文字数（例：68,500 文字）
- アーティストデート累計達成率（例：8週 / 12週中）

##### 2. 12週間の執筆文字数推移（グラフエリア）
- 横軸を第1週〜第12週（Week 1 〜 Week 12）とし、縦軸を各週の合計文字数、または平均文字数とした棒グラフ（またはラインチャート）を表示。
- まだ到達していない未来の週は、グラフ上で「未到達（Pending）」または非表示として扱う。

##### 3. 日次アクティビティヒートマップ
- GitHubのコントリビューショングラフ（草）のように、12週間×7日間（＝84マス）のマトリクスを表示。
- モーニングページを実行した日は文字数に応じて濃い色で塗りつぶされ、実行していない日は空白/薄いグレーで表示されることで、自身の「12週間の記載のムラやリズム」を視覚的に捉えられるようにする。
- **本日のセル表示（v2.4 新規）**: 現在の日付（本日）のセルを視覚的に特定できるようにする。
  - **判定**: クライアントのシステム日付と各セルの `daily_activity[].date` を突合し、一致するセルを「本日」とする（新規API不要）。
  - **表示方法（本バージョンの方針）**: 本日セルの**枠線をアクセントカラーで強調**（リング）し、あわせて本日セルのホバー時ツールチップに「**今日**」を明記する。塗りつぶし色（文字数に応じた濃淡）は従来通りで、記録の有無に関わらず本日が分かるようにする。
  - 本日がプログラム期間（84日）外の場合は、いずれのセルも「本日」として強調しない。

##### 3-2. 日次データの選択と詳細確認（v2.3 新規）
- ヒートマップ（カレンダー）やリスト表示されている各日の実績を、**ラジオボタンのように単一選択**（アクティブ状態を切り替え可能）できる UI にする。1つの日付を選択すると、それまでの選択は解除される。
- 日付を選択すると「**詳細**」ボタン（またはリンク）が出現し、押下することで該当日のモーニングページ内容を確認できる。
- **詳細の表示方法（本バージョンの方針）**: 既存の `/morning-page?date=YYYY-MM-DD` 画面へ遷移して閲覧・編集する（spec3 で実装済みの日付指定モーニングページ画面を再利用）。過去日の場合はモーニングページ画面側のルール（過去ページ編集の確認・保存後 `/home` リダイレクト）に従う。

##### 4. アーティストデートの軌跡（v2.3 表記調整）
- 第1週〜第12週それぞれのアーティストデート実施状況を、直感的な記号で一覧表示する。
- **実施した週**: 「**〇**（マル / アクセントカラー）」を表示。
- **未実施の週**: 「**✕**（バツ / グレー）」を表示。
- ひと目で達成状況が把握できるよう、リストまたはグラフの軸に沿って週ごとに配置する（`went_out` / `excited` をそれぞれ 〇/✕ で表現）。

## 6. 共通コンポーネント

v2.3 / v2.4 で追加・共通化を推奨するコンポーネント。

### 6.1 トースト / スナックバー（v2.3 新規）
- 画面下部または上部に一時的な通知を表示し、数秒後にフェードアウトする再利用可能なコンポーネント。
- 主な利用箇所: モーニングページ保存後の `/home` リダイレクト直後（「モーニングページを保存しました」）。
- メッセージ・表示時間（既定 3 秒程度）をプロパティで受け取れるようにする。

### 6.2 プログラム完遂モーダル（v2.3 新規）
- 12週間プログラム完遂（経過日数 Day 84 到達）時に表示する祝福モーダル。
- 紙吹雪（Confetti）演出（例: `canvas-confetti` 等のライブラリ、または CSS アニメーション）を含む。
- `prefers-reduced-motion` を尊重し、モーション低減設定時は演出を控えめにする。
- 「閉じる」でホーム画面に戻る。本バージョンでは到達後ログインのたびに表示する（表示済みフラグ保存なし）。

### 6.3 達成セレブレーション表現（v2.4 新規）
- モーニングページ 7/7 / アーティストデート完了時に、該当カードへ適用する祝福スタイル（薄ゴールド背景＋豪華アイコン）。
- カードコンポーネントに「達成（celebrating）」状態のプロパティを持たせ、背景・アイコンを切り替えられるようにすることを推奨。
- 派手なアニメーションは伴わない静的表現を基本とし、達成状態が続く限りホーム表示のたびに適用する（状態ベース）。

### 6.4 本日インジケーター（v2.4 新規）
- ホーム週次（バッジ「今日」＋リング）／統計ヒートマップ（枠線アクセント＋ツールチップ「今日」）で本日を示す共通の考え方。
- 「本日」判定はクライアントのシステム日付と対象セルの日付（`date`）の突合で行う（新規API不要）。
