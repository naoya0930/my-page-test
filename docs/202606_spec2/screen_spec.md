## 1. 画面一覧・ルーティング

| パス | 画面名 | 概要 | 認証制限 |
| :--- | :--- | :--- | :--- |
| `/` | LP (ランディングページ) | アプリの名称・概要説明、ログインへの誘導 | 不要 |
| `/login` | ログイン画面 | Supabase Authを利用したサインイン画面 | 不要 |
| `/home` | ユーザーホーム画面 | 現在・過去・未来の週次進捗確認、各タスクへのリンク | **必須** |
| `/morning-page` | モーニングページ画面 | 毎日の朝のノート（テキスト）入力・文字数カウント・保存 | **必須** |
| `/artist-date` | アーティストデート画面 | 週次のアーティストデート実施チェック・保存 | **必須** |
| `/statistics` | 統計画面（新規） | 12週間プログラム全体の入力傾向・アクティビティ可視化 | **必須** |

---

## 2. 画面遷移図（フロー）

1. **未認証ユーザーのアクセス**
   - どのアドレスにアクセスしても、`/` または `/login` 以外はフロントエンドで捕捉し、`/login` へリダイレクト。
2. **ログインフロー**
   - `/login` で認証成功 ➔ `/home` へ遷移。
3. **ホームからの遷移**
   - `/home` ➔ [モーニングページ入力] ➔ `/morning-page`
   - `/home` ➔ [アーティストデート入力] ➔ `/artist-date`
   - `/home` ➔ [統計を見る] ➔ `/statistics`
4. **タスク完了後のフロー**
   - `/morning-page` ➔ [保存] ➔ 成功時：状態を即時更新して `/home` へ戻る（または同画面で完了通知後、同期を維持）
   - `/artist-date` ➔ [保存] ➔ 成功時（安全に保管された場合）：**自動的に `/home` へ遷移する**（追加要件）
5. **共通ナビゲーション**
   - 認証後の全画面において、ヘッダー等の共通コンポーネントから `/home` および `/statistics` への相互遷移が可能。

## 3. 各画面詳細仕様

### 3.1 LP (ランディングページ)
- **表示内容:**
  - アプリの名称と12週間プログラムをサポートする旨の簡潔な説明。
  - 書籍本文の具体的な引用は行わず、抽象的な機能紹介にとどめる。
  - ブラウザ通知 / リマインダー機能の紹介文。
- **UI/アクション:**
  - 中央に目立つ「ログインへ」のCTAボタンを配置。クリックで `/login` へ遷移。

### 3.2 ログイン画面
- **表示内容:**
  - Supabase Authと連携したログインフォーム（メール/パスワード、またはソーシャルログイン）。
- **挙動・セキュリティ:**
  - 認証成功時、アクセストークンをフロントエンド（状態/ストレージ）に保持。
  - ログインエラー時は、フォーム下部にエラーメッセージ（「メールアドレスまたはパスワードが違います」等）を即時表示。

### 3.3 ユーザーホーム画面 (`/home`)
ユーザーがプログラムの進捗を確認し、その日のアクションを起こすための中心的な画面です。

#### ① 週次表示とタイムトラベル（左右スワイプ）機能（追加要件）
- **初期表示:**
  - アクセス時、ユーザーの開始日と現在のシステム日付から算出された**「現在の自身の経過週（例：第3週目）」**をデフォルトで中央に表示する。
- **左右スワイプ / ナビゲーション:**
  - 画面を左右にスワイプ（または左右の矢印ボタン「＜」「＞」をクリック）することで、過去の週（第1週〜現在未満）や未来の週（現在より上〜第12週まで）の状態に表示を切り替えられる。
- **「いまではないもの」を見ている際の視覚的演出（追加要件）:**
  - 現在の表示週が「現在の本人の経過週」と異なる場合、UIの色調を明確に変更し、過去や未来のデータを見ている（タイムトラベル中である）ことをユーザーに認識させる。
  - *演出案:*
    - **過去の週を表示中:** 画面全体の色彩を「セピア」または「モノトーン（グレースケール）」に変更。各入力ボタンは非活性（閲覧のみ）。
    - **未来の週を表示中:** 画面全体に「半透明のオーバーレイ」または「ソフトブルーのグラデーション」を適用し、未来のプレースホルダーであることを明示。タスクはロック状態とする。
    - **現在週を表示中:** アプリケーションの通常ブランドカラー（高コントラスト、アクティブ表示）で表示。

#### ② モーニングページ・日次実行ステータス（追加要件）
- **表示デザイン:**
  - 今週（表示中の週）の7日間において、何日目に実行し、何日目に実行していないのかがひと目で直感的にわかるデザインを採用する。
  - *デザイン案（ドットグリッド / カレンダーマトリクス）:*
    - 横一列に7つの円（ドット）またはカードを配置（Day 1 〜 Day 7）。
    - **実行済みの日の表示:** 鮮やかな成功カラー（例: ターコイズグリーン、チェックマーク付きの塗りつぶし）で表示。
    - **未実行（かつ過去・当日）の日の表示:** グレーアウト、または破線の枠線のみで表現し、スキップされたか未着手であることを明示。
    - 各ドットの下部には、その日に執筆した「文字数」を小さく表示する。
- **リンク:**
  - 当日のドット、または「今日のモーニングページを書く」ボタンから `/morning-page` へ遷移。

#### ③ アーティストデート進捗表示
- **表示内容:**
  - 今週分の実施状況をチェックリスト形式で表示。
    - 「今週、アーティストデートに出かけた」
    - 「今週、わくわくした出来事があった」
- **リンク:**
  - クリックで `/artist-date` へ遷移。

#### ④ バグ修正：編集後のステータス同期（追加要件）
- **修正要件:**
  - モーニングページを新規作成、または過去の記述を編集・保存した後にホーム画面に戻った際、進捗状態（`Progress` テーブルおよびフロントエンド状態）が正しく反映されず、タスクが「未完了」のままになってしまうバグを完全に修正する。
- **実装上の挙動:**
  - 保存処理が成功した段階で、フロントエンドの状態管理（React Context / Redux / SWR / TanStack Query等）のキャッシュを強制的に無効化（Invalidate）するか、最新の Progress API を再フェッチ（Refetch）する。
  - ホーム画面のコンポーネントがマウントされた際、常に最新のDB状態（当日分の `MorningPages` レコードの有無）を参照し、「完了」ステータスへの切り替えを遅延なく行う。

### 3.4 モーニングページ画面 (`/morning-page`)
- **表示内容:**
  - 広範なテキスト入力欄（マルチラインのTextArea、フォーカス時に枠線強調）。
  - **リアルタイム文字数カウント:** 改行、空白スペースを含むすべての入力文字数をミリ秒単位でクライアントサイドでカウントし、「〇〇文字」とリアルタイム表示。
  - 「保存」ボタンおよび「ホームへ戻る」リンク。
- **バリデーション & 挙動:**
  - テキストが空（0文字）の場合は、保存ボタンを非活性化するか、押下時に「内容を入力してください」と警告を表示してAPI送信をブロックする。
  - 保存ボタン押下時、バックエンド（Cloudflare Workers）の `POST /api/morning-pages` へ `entry_date`, `content`, `character_count` を送信。成功時はトースト通知を表示し、ステータスを即時同期（3.3-④を参照）。

---

### 3.5 アーティストデート画面 (`/artist-date`)
- **表示内容:**
  - 2つの独立したトグルスイッチまたはチェックボックス。
    1. 「今週、アーティストデートに出かけた」 (`went_out`: boolean)
    2. 「今週、わくわくした出来事があった」 (`excited`: boolean)
  - 「保存」ボタン。
  - 画面下部に、直近に保存されたステータス（同期時刻など）を表示。
- **保存時の遷移挙動の変更（追加要件）:**
  - ユーザーが「保存」ボタンを押下する。
  - バックエンドAPI（`POST /api/artist-dates`）への送信が走り、データベース（D1）への書き込みが正常に完了して `200 OK` が返却される。
  - **安全に保管されたことが確認できた場合、画面内に留まらず、自動的にユーザーホーム画面（`/home`）にリダイレクト（遷移）する。**
  - 保存失敗時は、エラーメッセージ「保存に失敗しました。再度お試しください。」を表示し、画面を遷移させずにユーザーの再試行を促す。

## 4. 統計画面 (`/statistics`) 【新規追加画面】

ユーザーが12週間のプログラムにおいて、どのような軌跡をたどってきたか（継続の証、自己の思考パターンの変化）を定量的に振り返るための画面です。

- **表示内容:**
  1. **12週間総合サマリーカード:**
     - モーニングページの総執筆日数（例：52日 / 84日中）
     - 累計総文字数（例：68,500 文字）
     - アーティストデート累計達成率（例：8週 / 12週中）
  2. **12週間の執筆文字数推移（グラフエリア）:**
     - 横軸を第1週〜第12週（Week 1 〜 Week 12）とし、縦軸を各週の合計文字数、または平均文字数とした棒グラフ（またはラインチャート）を表示。
     - まだ到達していない未来の週は、グラフ上で「未到達（Pending）」または非表示として扱う。
  3. **日次アクティビティヒートマップ:**
     - GitHubのコントリビューショングラフ（草）のように、12週間×7日間（＝84マス）のマトリクスを表示。
     - モーニングページを実行した日は文字数に応じて濃い色で塗りつぶされ、実行していない日は空白/薄いグレーで表示されることで、自身の「12週間の記載のムラやリズム」を視覚的に捉えられるようにする。
  4. **アーティストデートの軌跡:**
     - 第1週〜第12週それぞれの「出かけた (`went_out`)」「わくわくした (`excited`)」の達成状況をアイコンのオン/オフで一覧表示。

## 5. バックエンドAPIおよびデータモデルへの影響

仕様変更に伴い、以下のデータ・ロジックの保証が必要です。

1. **`GET /api/progress` の拡張:**
   - 特定の `week_number` をパラメータとして受け取り、その週に紐づく `Progress` 状況および `MorningPages`（7日間分の日次有無）、`ArtistDates` のフラグを一括で返却する設計にする（スワイプ時の高速描画のため）。
2. **`GET /api/statistics` の新設:**
   - ユーザーIDに紐づくすべての過去ログを走査し、週ごとの文字数集計、総日数、各フラグの成立数を集計してフロントエンドに返す軽量な集計クエリ（SQLの `GROUP BY week_number` 等を活用）を Cloudflare Workers 上に実装する。
3. **トランザクション/更新トリガーの適正化:**
   - `MorningPages` の保存・更新時に、該当する週の `Progress.morning_page_done` を即座に評価・更新するロジックをバックエンド側でも担保し、フロントエンドへの一貫したデータ返却を行う。
